# Kaifuu Fixture Policy

KAIFUU-000 defines how engine fixtures and reference implementations are
sourced before parallel adapter work begins. It extends
[fixtures-and-corpora.md](fixtures-and-corpora.md) and
[testing-standard.md](testing-standard.md) for real engine adapters.

Kaifuu is a tool for extraction, patching, verification, and `.kaifuu` delta
packages over user-provided project files. This policy does not claim support
for every commercial title and does not authorize redistribution of third-party
assets. Broad production support for every protected commercial variant is not
an alpha requirement. Local-only encrypted corpus triage, archive/encryption
detection, key-profile schema, secret redaction, platform-assisted helper
boundaries, and engine-specific encrypted research slices are alpha requirements.
See [kaifuu-key-discovery.md](kaifuu-key-discovery.md). When provenance,
licensing, or support scope is unclear, the adapter node stays blocked or
research-only until the project owner resolves the scope.

## Adapter Readiness Record

Every engine adapter needs a tracked readiness record before implementation
starts. The record may live in an adapter README, a future
`docs/kaifuu-adapters/<engine>.md` file, or another tracked document while the
crate is still being created. A public fixture manifest may reference that
readiness record, but the manifest itself is not expected to carry the full
record under the current `fixtures/public/manifest.schema.json` contract.
Do not rely on a public manifest as the sole readiness record unless a later
schema or playbook node extends manifests to include every required readiness
field. Chat notes are not durable records.

The record must include:

- Engine family, supported versions, and excluded versions.
- Fixture id, source class, source URL or local generation recipe, author or
  maintainer, license, and SHA-256 hashes for committed raw files.
- Support boundary: accepted file layouts, encodings, container formats, script
  variants, asset surfaces, and patch modes.
- Unsupported or gated boundary: encrypted, packed, compiled, obfuscated,
  platform-only, key-required, helper-required, or plugin-specific variants that
  must return semantic capability errors unless the adapter record explicitly
  supports the exact case.
- Reference implementation citations and license review outcome.
- Required round-trip artifacts and commands.
- Private corpus notes limited to redacted labels, aggregate stats, and local
  manifest hashes.

An adapter may be researched without a public fixture, but it must not move to
implementation or claim support until this record exists and names at least one
public round-trip fixture.

## Fixture Provenance

Public fixtures are the only fixtures allowed in public CI. Prefer synthetic
fixtures authored for this repository because they give adapter agents
deterministic coverage without carrying third-party content.

Allowed public fixture source classes:

- Synthetic fixture authored in-repo, with a README or manifest note describing
  who created it and what engine behavior it covers.
- Upstream sample project or example file that is explicitly redistributable
  under a reviewed open license.
- Public-domain or permissively licensed sample data with license evidence
  committed next to the manifest.
- Minimal binary or archive fixtures generated by a checked-in script when the
  script, source template, and generated bytes are all redistributable.
- Synthetic encrypted or packed fixtures when the archive, plaintext template,
  key, and generation script are all authored for this repository or otherwise
  redistributable. Public test keys must be visibly marked as fixture-only and
  must not resemble retail key material.

Disallowed committed fixture sources:

- Purchased games, trial builds, fan patches, retail scripts, extracted text,
  screenshots, audio, fonts, videos, save files, or asset dumps unless the
  owner has explicitly licensed those exact files for redistribution.
- Decompiled or unpacked commercial data used only because it is easy to find.
- Private corpus manifests that reveal local paths, story text, account data,
  order ids, license keys, or filenames that disclose restricted content.

Public fixture manifests must follow `fixtures/public/manifest.schema.json`,
include SHA-256 hashes and byte lengths, and validate with
`just fixtures-validate`. Engine fixtures should also document the engine
variant they represent, the adapter support boundary they exercise, and whether
the fixture is positive, negative, or round-trip evidence.

## Private Corpora

Private corpora stay under `fixtures/private-local/` and remain ignored by git.
They can be used for local benchmark credibility, gap analysis, and manual
smoke tests, but public CI and adapter acceptance cannot depend on them.

Private corpus reports may cite:

- Redacted corpus label.
- Acquisition class, such as purchased copy or licensed evaluation set.
- Aggregate stats and private manifest hash.
- Canonical hash-list hash, when filenames are safe or redacted as described in
  [fixtures-and-corpora.md](fixtures-and-corpora.md).
- Redacted key-profile ids, helper class, helper version, key-validation proof
  hashes, and readiness status.

Private corpus reports must not include raw strings, screenshots, extracted
tables, script snippets, local absolute paths, storefront account details, or
small aggregates that effectively disclose a specific source line. They must
also not include raw keys, local secret refs that reveal private paths, helper
memory dumps, decrypted scripts, or unredacted helper logs. When a private
corpus exposes a bug, reproduce it with a synthetic public fixture before
making the adapter acceptance test depend on it.

## License Review

Every committed fixture and every consulted reference implementation needs a
license review before code is ported, generated, or committed.

Record at least:

- Name, URL, upstream owner, and retrieval date or commit.
- SPDX license id when available, or the exact license text path when not.
- Whether redistribution, modification, generated artifacts, and attribution are
  permitted for the intended use.
- Whether the license has copyleft, non-commercial, no-derivatives, attribution,
  source-offer, or unclear terms that affect adapter code or fixtures.
- The decision: `allowed`, `allowed-with-attribution`,
  `behavior-only-clean-room`, `private-local-only`, or `blocked`.

If the license is missing, incompatible, or unclear, do not copy code, tables,
test data, constants, or comments from that reference into Kaifuu. Use only
public format documentation, independently authored notes, or synthetic
fixtures until the reference is cleared.

## Reference Implementation Citation

Reference implementations include open-source extractors, decompilers, archive
tools, engine docs, format notes, sample projects, and forum snippets used to
understand a format. Each adapter record must cite the references that shaped
its behavior.

Citation rules:

- Cite the exact repository URL, release, commit hash, file path, and relevant
  function or section when practical.
- Record the license review decision before porting starts.
- Distinguish direct porting, behavior comparison, test-oracle use, and general
  background reading.
- Cite in tracked docs or source comments near non-obvious format logic. Chat
  transcripts and local scratch notes are not sufficient.
- Keep attribution with generated code or generated fixtures when the upstream
  license requires it.

Porting rules:

- Prefer implementing from public format docs and fixture behavior.
- Do not paste or mechanically translate code from references unless the license
  is compatible with the target crate and attribution requirements are met.
- Do not copy large lookup tables, opcode maps, keys, signatures, or fixture
  bytes unless their license and provenance are explicit.
- For GPL, non-commercial, no-derivatives, unknown, or otherwise incompatible
  references, use a behavior-only or clean-room approach: one worker may write
  high-level format observations and fixture expectations, while the
  implementer writes code without copying expression from the reference.
- If a reference depends on bypassing encryption or platform secrets, keep that
  work out of the pure parser. Model it as a local key-discovery helper that
  returns structured redacted key-profile evidence, while pure adapters consume
  only resolved keys and public format parameters.

## Round-Trip Artifact Gate

Adapter implementation must not start from only a private corpus or a parser
idea. The first implementation slice for an engine needs public round-trip
artifacts checked into the repo or generated deterministically from checked-in
inputs.

Required artifacts before adapter work starts:

- Public fixture manifest with raw-file hashes and engine variant metadata.
- Positive source fixture containing representative dialogue, choices,
  speaker/name context, UI or database text when relevant, and protected engine
  markup.
- Expected bridge bundle or golden extraction artifact with schema version,
  source hashes, stable unit ids, patch references, and protected spans.
- Unchanged patch input and expected verify result. If the engine cannot be
  byte-identical after an unchanged patch, the record must name the normalized
  equivalence rule and the fields that may change.
- Translated patch input and expected patch result in a temp output directory.
- At least one negative fixture or synthetic detector case for unsupported
  encrypted, compiled, packed, or unknown variants.
- Synthetic encrypted/minimal archive fixtures with public test keys when an
  adapter or detector claims key-profile behavior.
- Archive/encryption matrix fixtures may be temp-directory builders in Rust
  tests when the committed bytes would otherwise imply real engine support.
  They must use synthetic marker bytes and assert aggregate evidence output,
  not raw source filenames, keys, decrypted text, helper dumps, or local paths.
- Commands that run through the shared round-trip harness once KAIFUU-003
  exists, or adapter-local commands until that harness lands.

Adapters that cannot provide these artifacts stay research-only. They may add
format notes or fixture requirements, but they should not add production parser
claims.

## Semantic Capability Errors

Unsupported engine variants must fail with semantic capability errors, not raw
I/O errors, panics, silent skips, or corrupted partial output. A semantic error
is a structured failure that tells Itotori, Utsushi, CI, and adapter agents what
capability is missing.

Future Kaifuu adapter errors should carry these fields when the adapter can
detect them:

- `errorCode`: stable code from the table below.
- `engine`: engine family detected or requested.
- `adapter`: adapter crate or registry id.
- `detectedVariant`: format, version, archive type, or script class.
- `assetRef`: repo-relative or profile-relative asset reference when safe.
- `requiredCapability`: capability needed to proceed.
- `supportBoundary`: short text or doc link explaining the accepted variants.
- `remediation`: user action, such as "extract archive first", "provide a key
  profile", or "use plaintext .rpy sources".

| Error code                                | Use when                                                                   |
| ----------------------------------------- | -------------------------------------------------------------------------- |
| `kaifuu.unsupported_variant.encrypted`    | Input is encrypted, protected, or requires unsupported key discovery.      |
| `kaifuu.unsupported_variant.compiled`     | Input is compiled bytecode or binary script the adapter cannot rebuild.    |
| `kaifuu.unsupported_variant.packed`       | Input is inside an archive or bundle that this adapter does not unpack.    |
| `kaifuu.missing_capability.key_profile`   | The adapter can parse with user-provided keys, but no key profile exists.  |
| `kaifuu.missing_key_material`             | A key profile exists, but required local key material cannot be resolved.  |
| `kaifuu.helper_unavailable`               | The requested local, Wine, Windows, or remote helper is not available.     |
| `kaifuu.key_validation_failed`            | Supplied or discovered key material fails deterministic local validation.  |
| `kaifuu.protected_executable_unsupported` | Helper analysis detected protection that the current helper cannot handle. |
| `kaifuu.secret_redacted`                  | A result intentionally omitted secret-bearing fields from logs or reports. |
| `kaifuu.unsupported_variant.obfuscated`   | Input is intentionally obfuscated or generated beyond the support scope.   |
| `kaifuu.unsupported_surface.asset_text`   | A visible text surface is detected but patching that asset type is absent. |
| `kaifuu.unknown_engine_variant`           | The fingerprint is close to an engine family but not a profiled variant.   |

Extraction may still emit partial capability reports if no files are modified.
Patching must fail before writing output when a required source asset is an
unsupported encrypted, compiled, or packed variant.

## Agent Development Expectations

Parallel adapter agents should treat this policy as a start gate:

- Read this policy, the target roadmap node, and existing fixture/corpus docs
  before touching adapter code.
- Add or update the adapter readiness record first.
- Add public positive and negative fixtures before broad parser work.
- Keep private corpus evidence local and summarize only hashes or aggregate
  facts in tracked reports.
- Keep support claims scoped to the fixture profile that round-trips in CI.
- Return semantic capability errors for recognized unsupported variants.
- For encrypted inputs, keep public CI public-fixture-only while making
  private-local encrypted validation easy to run and safe to summarize.
- Leave roadmap follow-up recommendations as P2/P3 nodes when support requires
  production binary patching, full encrypted archive patching, runtime
  validation, or unavailable fixtures.
